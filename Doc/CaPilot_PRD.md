## 1. 问题陈述

### 1.1 核心问题

开发者同时运行多个 AI Agent（编码代理、研究代理、分析代理、写作代理等）时，面临严重的 I/O 瓶颈：**一套键盘 + 鼠标 + 单个显示器无法高效管理 5~20+ 个并发的 Agent 会话。**

每切换一个 Agent 都需要 Alt+Tab 切窗口、滚动日志、找回上下文、重新聚焦。高频切换带来巨大的认知负担——这恰恰是 AI Agent 本想帮用户解决的问题。

更关键的是，**双手已经绑在主键盘上时**，任何额外的鼠标操作、窗口切换、文本输入都会打断编码流。理想的方案是让开发板同时承担多面手角色：**触摸板代替鼠标、语音代替键盘输入、体感代替快捷键**——不离开主工作区就能完成所有 Agent 管理。

### 1.2 谁在承受这个问题

- **独立开发者 / 自由职业者** — 同时使用多个 AI 编码 Agent 完成不同功能模块，需要在 Agent 间频繁切换
- **多 Agent 调度的工程师** — 运行多个 Agent 处理不同的任务（编码、调试、文档、测试），需要实时监控和指挥
- **AI 工作流重度用户** — 同时使用多个 AI 工具（Claude + ChatGPT + Cursor + 自定义 Agent），缺少统一指挥界面

### 1.3 不解决的代价

| 维度 | 影响 |
|------|------|
| **时间损失** | 每次 Agent 切换平均 5~15 秒，每天累积 30~60 分钟浪费在上下文切换上 |
| **认知负荷** | 高频 Alt+Tab 导致"注意力残留"，打断深度工作状态（Flow State） |
| **错过关键事件** | Agent 输出的关键变更、报错、结果在日志洪流中被淹没 |
| **多 Agent 协作困难** | 无法直观感知各 Agent 的"忙/闲/有结果/需确认"状态 |
| **效率上限** | 现有工具链限制了用户同时利用的 Agent 数量上限 |
| **物理空间受限** | 桌面空间有限，额外放一个键盘/触摸板不现实，需要多功能集成 |

### 1.4 证据支撑

- 用户（HaChi）自身在同时管理 DesTopPet、人形机器人、博客等多个项目时，需要在多个 Agent 间反复切换
- 当前工作流中，Agent 输出分散在多个终端窗口、浏览器标签页、聊天窗口中，缺乏统一的物理指挥界面
- 多 Agent 场景正随着 LLM 发展快速普及——这不是一个小众需求，而是 AI 原生工作模式的刚需
- 立创实战派 ESP32-S3 开发板已集成触摸屏、麦克风阵列（ES7210+ES8311）、6 轴陀螺仪（QMI8658），硬件基础充分

## 2. 目标

### 2.1 产品目标

| # | 目标 | 衡量标准 |
|---|------|----------|
| G1 | **将 Agent 管理操作时间降低 80%** | 单次操作（切换/指令/查看状态）从 5~15 秒降至 ≤2 秒 |
| G2 | **实现"零窗口切换"的 Agent 管理** | 用户无需离开主工作窗口即可管理所有 Agent |
| G3 | **提供即时的 Agent 状态感知** | 用户在一瞥之间（≤1 秒）了解所有 Agent 的当前状态 |
| G4 | **触摸板替代鼠标** | 屏幕触摸区域可直接模拟鼠标指针移动和点击，延迟 ≤50ms |
| G5 | **语音驱动 Agent 命令** | 通过开发板麦克风说出指令，转文字后发送到指定 Agent，端到端延迟 ≤3 秒 |
| G6 | **体感切换窗口** | 通过陀螺仪手势（如甩动/倾斜）切换显示器上的应用/终端窗口，成功率 ≥90% |
| G7 | **2 按键模式切换** | 开发板仅有 2 个物理按键，但通过模式切换（≥3 种模式）覆盖全部核心功能 |

### 2.2 用户目标

- **获得"指挥中心"式体验** — 一个物理设备 + 软件后端，像音乐混音台控制音轨一样控制 Agent
- **减少打断** — 不再因为翻找 Agent 输出而中断当前工作
- **获得状态安全感** — 随时知道每个 Agent 在干什么、有没有产出、需不需要我介入
- **双手不离主键盘** — 通过触摸板/语音/体感完成所有交互，减少在键盘和鼠标之间的手部移动

### 2.3 商业目标

- 为后续"CaPilot"外设产品线建立品牌认知和用户基础
- 通过 MVP 验证多 Agent 用户群体的付费意愿
- 收集用户行为数据指导硬件迭代方向
- 探索"多功能桌面伴侣"品类的可行性和市场空间

## 3. 非目标

| # | 非目标 | 原因 |
|---|--------|------|
| NG1 | **不做 Agent 本身** — CaPilot 是指挥外设，不是 Agent 运行平台 | Agent 运行在后端/云端，CaPilot 只负责调度交互 |
| NG2 | **不做通用输入设备** — 不替代主键盘/鼠标 | 已有成熟方案，CaPilot 聚焦多 Agent 管理场景，触摸板/语音是辅助输入而非替代主设备 |
| NG3 | **不做全功能显示屏** — 不追求高分辨率、高刷新率屏幕 | 核心是"一瞥即可"的状态显示 + 触控交互，不是工作主屏 |
| NG4 | **v1 不做远场语音识别** — 语音交互需靠近麦克风（≤30cm） | 近场语音更可靠，远场降噪复杂度高 |
| NG5 | **v1 不支持无线远离** — 需通过 USB 连接主机供电和通信 | 减少复杂度，聚焦核心体验，v2 考虑 BLE 无线模式 |
| NG6 | **不做 Agent 训练/调优** — 不涉及 Prompt 工程、RAG 知识库管理等 | 这些是 Agent 平台的事，CaPilot 专注"界面 + 调度" |
| NG7 | **v1 不做高级手势识别** — 触控板仅支持基础手势（移动/点击/滚动），不支持多指复杂手势 | 基础交互即可满足 MVP，复杂手势可软件端扩展 |

## 4. 用户故事

### P0 — MVP 必须有

| ID | 用户故事 | 优先级 |
|----|----------|--------|
| US-01 | **作为多 Agent 开发者**，我希望在 CaPilot 的屏幕上**一览所有正在运行的 Agent 及其状态**（运行中/已完成/报错/等待输入），这样我不用切窗口就能掌握全局 | P0 |
| US-02 | **作为双手不离键盘的开发者**，我希望**用手指在屏幕上滑动就能控制电脑鼠标**（触摸板模式），这样我不用伸手去够鼠标 | P0 |
| US-03 | **作为忙碌的开发中用户**，我想**对着设备说话来向 Agent 下达指令**（如"让编码 Agent 帮我写一个排序函数"），语音转文字后自动发送给指定 Agent | P0 |
| US-04 | **作为多窗口工作的开发者**，我想**甩动开发板或倾斜设备就能切换电脑上的窗口**（如终端→浏览器→IDE），这样我不用 Alt+Tab | P0 |
| US-05 | **作为有限物理按键的妥协者**，我希望**2 个按键在不同模式下承担不同功能**（如 Agent 模式下切换选中、触控板模式下切换左右键），模式当前在屏幕上清晰指示 | P0 |
| US-06 | **作为重度 Agent 用户**，我想通过**点击屏幕上的 Agent 卡片来切换控制焦点**，这样我可以在触摸板模式和 Agent 管理模式之间无缝流转 | P0 |
| US-07 | **作为需要并行工作的开发者**，我希望 Agent **完成后在设备上产生明显的视觉反馈**（LED 闪烁、屏幕高亮），这样我不会错过关键产出 | P0 |
| US-08 | **作为一名开发者**，我希望 CaPilot **通过 USB 即插即用**，安装配套软件后自动发现并连接，无需复杂配置 | P0 |

### P1 — 重要但可后置

| ID | 用户故事 | 优先级 |
|----|----------|--------|
| US-09 | **作为高级用户**，我想**自定义各模式下按键的功能映射**，这样我可以根据自己的工作流设定最适合的快捷键组合 | P1 |
| US-10 | **作为关注效率的用户**，我希望设备 **显示每个 Agent 的运行耗时和资源占用**，这样我能发现哪些 Agent 效率低需要优化 | P1 |
| US-11 | **作为多显示器用户**，我想用 CaPilot **快速将 Agent 的输出推送到指定显示器或桌面区域**，这样我能物理上"安排"每个 Agent 的工作位置 | P1 |
| US-12 | **作为语音交互用户**，我希望**在屏幕上看到语音转文字的实时预览**，确认后再发送，这样避免误识别导致错误指令 | P1 |

### P2 — 未来版本

| ID | 用户故事 | 优先级 |
|----|----------|--------|
| US-13 | **作为团队协作成员**，我想通过 CaPilot 查看**团队其他成员公开的 Agent 状态**，这样我能了解团队的工作进度 | P2 |

## 5. 需求

### 5.1 硬件需求

#### H1 — 显示与触控系统

| ID | 需求描述 | 验收标准 |
|----|----------|----------|
| H1.1 | 搭载 ≥2.0 英寸 LCD 显示屏，分辨率 ≥320x240，支持触摸 | 屏幕可清晰显示 8 个 Agent 的状态图标 + 文字标签；触摸响应延迟 ≤50ms |
| H1.2 | 屏幕刷新率 ≥30fps | 状态切换动画流畅无卡顿 |
| H1.3 | 屏幕亮度 ≥300nit，支持室内阅读 | 正常室内光线下清晰可见 |
| H1.4 | 触摸屏支持电容式多点触控（至少 2 点），用于触控板模式 | 支持单指移动/点击 + 双指滚动 |

> **当前硬件基础：** 立创实战派 ESP32-S3 搭载 FT5x06 电容触摸控制器，支持 2 点触控，已验证可用。详见 `Hardware/dev_boards/lceda_esp32_s3_practical_board/02-docs/02-chip-manuals/`。

#### H2 — 物理交互

| ID | 需求描述 | 验收标准 |
|----|----------|----------|
| H2.1 | **2 个原生物理按键**（BOOT + RST），通过长按/短按/组合实现模式切换 | 短按 = 当前模式主功能；长按 = 模式切换；组合（同时按）= 回默认模式 |
| H2.2 | 按键功能随当前活跃模式自动切换，按键的功能映射在屏幕上实时显示 | 用户看一眼屏幕就知道当前按键是干什么的 |
| H2.3 | RGB LED 指示灯（≥3 个），表示模式/连接/通知状态 | 不同模式有不同颜色标识（如蓝=Agent 模式，绿=触控板模式，紫=语音模式） |

#### H3 — 音频输入

| ID | 需求描述 | 验收标准 |
|----|----------|----------|
| H3.1 | 通过板载麦克风（ES7210 音频编解码器）采集用户语音 | 采样率 ≥16kHz，16bit，单声道 |
| H3.2 | 音频数据通过 USB 串口流式传输到主机端进行语音识别 | 音频流延迟 ≤500ms（采集→发送→识别→返回文本） |
| H3.3 | 语音交互在近场（≤30cm）环境下识别准确率 ≥90% | 安静环境下测试 |

> **当前硬件基础：** 立创实战派配备 ES7210（4 通道 ADC，用于麦克风阵列）和 ES8311（音频编解码 + 功放），已验证录音和播放功能。

#### H4 — 姿态传感（陀螺仪）

| ID | 需求描述 | 验收标准 |
|----|----------|----------|
| H4.1 | 通过 QMI8658 6 轴 IMU 检测设备姿态和运动 | 支持加速度 ±2g/±4g/±8g/±16g，陀螺仪 ±125°/±250°/±500°/±1000°/±2000°/s |
| H4.2 | 预置至少 3 种体感手势：**甩动（Flick）**切换窗口、**左倾/右倾**切换桌面、**拍击（Tap）**唤醒/切换模式 | 手势识别成功率 ≥90%，误触发率 ≤5% |
| H4.3 | 姿态数据通过 USB 串口上报手势事件，不在设备端做复杂姿态解算 | 仅上报原始手势事件（如 `{"type":"gesture","gesture":"flick_left"}`），具体窗口切换逻辑由主机端执行 |

> **当前硬件基础：** QMI8658 6 轴姿态传感器已在开发板上集成，`02-attitude` 例程验证了姿态数据读取，详见 `Hardware/dev_boards/lceda_esp32_s3_practical_board/01-examples/02-attitude/`。

#### H5 — 连接与供电

| ID | 需求描述 | 验收标准 |
|----|----------|----------|
| H5.1 | USB-C 接口，同时提供供电和数据通信 | 插上电脑即可使用，无需额外电源 |
| H5.2 | USB 串口（CDC ACM）双向通信，波特率 115200 | 传输触摸/音频/手势/按键事件到主机，接收 Agent 状态和指令 |
| H5.3 | (v2) 支持 BLE 无线模式 | 与有线模式无缝切换 |

### 5.2 固件需求

#### F1 — 模式管理系统（新核心）

| ID | 需求描述 | 验收标准 |
|----|----------|----------|
| F1.1 | 定义 ≥3 种操作模式：**Agent 模式** / **触控板模式** / **语音模式** | 模式切换动画（≤300ms），屏幕顶部显示当前模式图标 |
| F1.2 | 模式切换方式：长按 BOOT 键（>1.5s）循环切换；或通过触控屏幕上的模式图标直接切换 | 所有切换方式均可靠 |
| F1.3 | 每个模式下，2 个物理按键的功能和屏幕布局随之切换 | 按键功能映射表见 F2；屏幕 UI 按模式重新布局 |
| F1.4 | 模式持久化：设备断电后恢复上次使用的模式 | NVS 存储模式状态 |

**按键功能映射表（2 键 × 3 模式）：**

| 模式 | BOOT 按键 | RST 按键 | 长按 BOOT |
|------|-----------|----------|-----------|
| **Agent 模式** | 选中下一个 Agent / 确认 | 返回上一级 / 取消 | 切换到下一个模式 |
| **触控板模式** | 鼠标左键 | 鼠标右键 | 切换到下一个模式 |
| **语音模式** | 开始/停止录音 | 确认发送 / 取消 | 切换到下一个模式 |

#### F2 — 显示系统（LVGL）

| ID | 需求描述 | 验收标准 |
|----|----------|----------|
| F2.1 | **Agent 模式 UI**：Agent 列表视图，显示每个 Agent 的名称和当前状态，支持触控点击选中 | 一屏最多显示 6~8 个 Agent，支持触控滚动 |
| F2.2 | **触控板模式 UI**：全屏触控区域，顶部显示模式指示条、鼠标坐标信息 | 手指触摸区域覆盖整个屏幕，排除顶部状态条 |
| F2.3 | **语音模式 UI**：中央显示波形动画（音频可视化）+ 语音转文字实时预览 + 确认/取消按钮 | 波形随音量实时变化，文字预览更新延迟 ≤500ms |
| F2.4 | 状态指示条：始终在屏幕顶部，显示当前模式、连接状态、总 Agent 数 | 所有模式下均可见 |
| F2.5 | 动画过渡：模式切换和 Agent 状态变化时有伴随动画 | 平滑过渡，不突兀 |

#### F3 — 触控板子模块

| ID | 需求描述 | 验收标准 |
|----|----------|----------|
| F3.1 | 在触控板模式下，全屏捕获触摸事件（绝对坐标），映射为鼠标相对位移上报 | 触摸起始点记录，手指移动时上报相对位移量（dx, dy） |
| F3.2 | 单指点击 = 鼠标左键单击，双指点击 = 鼠标右键 | 点击检测算法：触摸 → 抬起的时长 ≤300ms 且位移 ≤10px |
| F3.3 | 双指垂直滑动 = 滚轮滚动 | 滚动速度与手指移动速度成正比，有惯性效果 |
| F3.4 | 触摸事件通过 HID 鼠标协议或自定义 JSON 上报主机 | 主机端守护进程接收后转换为系统鼠标/滚轮事件 |
| F3.5 | 支持触摸板加速度曲线：慢移精细、快移大范围 | 指针移动速度符合"手指位移 × 速度因子"曲线 |

#### F4 — 语音处理子模块

| ID | 需求描述 | 验收标准 |
|----|----------|----------|
| F4.1 | 在语音模式下，按下 BOOT 键开始录音，从 ES7210 采集音频并缓存 | 缓存大小 ≤64KB（约 2 秒 16kHz 16bit 音频） |
| F4.2 | 实时将音频数据通过 USB 串口流式传输到主机端 | 分包发送，每包 ≤512 字节，带序列号 |
| F4.3 | 主机端接收音频流后调用 STT 引擎（如 Whisper 本地模型或云端 API）转换为文本 | 端到端延迟 ≤3 秒（从按下录音到文本返回） |
| F4.4 | 屏幕实时显示语音转文字预览 | 推送文本实时更新在屏幕中央区域 |
| F4.5 | 用户确认（RST 短按）后将文本作为指令发送到当前选中的 Agent | 取消（RST 长按）则丢弃文本 |
| F4.6 | 板上提供 LED 反馈：录音时 LED 常亮闪烁，识别中 LED 呼吸，完成时 LED 短暂绿灯 | 用户通过 LED 状态了解语音处理阶段 |

#### F5 — 陀螺仪手势子模块

| ID | 需求描述 | 验收标准 |
|----|----------|----------|
| F5.1 | 在任意模式下，通过 QMI8658 持续监测设备姿态 | 采样率 ≥100Hz |
| F5.2 | 预置手势：**甩动（Flick）** = 检测到 Z 轴角速度突变（>阈值 400°/s）→ 上报 flick 事件 | 成功触发率 ≥90%，假触发 ≤5% |
| F5.3 | 预置手势：**左倾/右倾** = 设备绕 Y 轴倾斜 >30° 保持 >500ms → 上报 tilt_left / tilt_right | 用于桌面切换或应用切换 |
| F5.4 | 预置手势：**拍击（Tap）** = Z 轴加速度突变（>3g）→ 上报 tap 事件 | 用于唤醒或快速切换模式 |
| F5.5 | 手势事件通过串口 JSON 上报，格式：`{"type":"gesture","gesture":"flick_right","confidence":0.95}` | 带置信度分数，主机端可据此判断是否执行 |

#### F6 — 通信协议

| ID | 需求描述 | 验收标准 |
|----|----------|----------|
| F6.1 | 通过 USB 串口（CDC ACM）与主机双向通信 | 波特率 115200，数据格式 JSON + 二进制音频流 |
| F6.2 | 主机→设备：Agent 状态推送协议（JSON） | 格式 `{"type":"agent_status","agents":[{"id":"a1","name":"编码Agent","status":"running"}]}` |
| F6.3 | 设备→主机：按键/触摸/手势事件上报（JSON） | 格式 `{"type":"key_event","key":"boot","action":"press","mode":"agent"}` |
| F6.4 | 设备→主机：触摸板事件上报（JSON） | 格式 `{"type":"touchpad","dx":12,"dy":-5,"action":"move"}` |
| F6.5 | 设备→主机：音频流数据（二进制分包） | 格式：包头（4 字节长度 + 1 字节 seq） + 音频数据 |
| F6.6 | 心跳机制：每 5 秒双向心跳，超时 15 秒视为断连 | 断连时屏幕显示"连接断开"提示，LED 变红 |

#### F7 — 状态机

| ID | 需求描述 | 验收标准 |
|----|----------|----------|
| F7.1 | 支持的 Agent 状态：idle / running / waiting_input / completed / error | 每种状态有独立的图标、颜色和 LED 指示 |
| F7.2 | 未读通知标记：Agent 有新产出时标记为 unread | Agent 卡片上有小红点/高亮，LED 闪烁 |
| F7.3 | 设备状态：connected / disconnected / error / voice_recording / processing | 屏幕顶部状态条实时显示 |

| ID | 需求描述 | 验收标准 |
|----|----------|----------|

| S3.4 | 用户确认后，将文本作为指令通过 Agent 连接器发送到指定 Agent | 支持自然语言指令解析（如"让编码 Agent 帮我重构这个函数"→ 定位到编码 Agent + 发送指令文本） |

## 6. 成功指标

### 6.1 领先指标（上线后 1~4 周评估）

| 指标 | 目标值 | 衡量方式 |
|------|--------|----------|
| **激活率** | ≥70% 的用户在首次连接后 7 天内持续使用 | 守护进程记录设备在线时长 |
| **日使用时长** | 重度用户日均使用 ≥4 小时 | 设备活跃时间统计 |
| **模式使用分布** | 三种模式均有 ≥30% 的使用占比 | 固件统计各模式累计使用时间 |
| **触摸板响应延迟** | 触摸 → 鼠标移动的端到端延迟 ≤50ms | 守护进程日志测量 |
| **语音识别延迟** | 按下录音 → 文本返回 ≤3 秒 | 端到端计时 |
| **手势识别成功率** | 体感手势识别率 ≥90%，误触发 ≤5% | 固件统计 + 用户反馈 |
| **错误率** | 通信断连、状态不同步等错误率 ≤1% | 守护进程错误日志统计 |
